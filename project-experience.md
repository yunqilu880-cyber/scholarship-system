# 项目经历：奖学金智能评选系统

## 项目概况

**项目名称**：浙江工商大学奖学金智能评选系统（Scholarship Evaluation System）

**项目定位**：面向高校教务场景的自动化奖学金评选工具，替代传统人工审核流程，实现从学生数据录入、奖项规则配置到评选结果输出的一站式管理。

**状态**：概念验证原型（未正式实施）

**角色**：独立全栈开发

**技术栈**：React 19 + TypeScript 6 + Vite 8 + Tailwind CSS 4 + React Router 7 + Lucide React

---

## 项目背景

高校奖学金评选历来依赖教务老师人工核对成绩、志愿时长、体测分数等多维度数据，再结合互斥规则（如"国家奖学金与校一等奖学金不可兼得"）进行名额分配。一个中等规模的学院每学年需要处理数百名学生的评选工作，流程繁琐且易出错。

我以**浙江工商大学**的真实奖学金评选流程为蓝本，设计并开发了这套自动化评选系统，目标是将原本需要数天的人工审核压缩到分钟级。

---

## 核心功能

### 1. 学生数据管理
- 支持学生基本信息、学业成绩（GPA）、德育分、实践分、体测分、加分项等**13个维度**的数据录入
- 提供单条添加/编辑 + **CSV批量导入**两种数据录入方式
- 按姓名/学号搜索、按学院筛选，实时过滤

### 2. 奖项与规则配置
- 多级奖学金配置（国家级/校级/院级），每项可独立设定金额、名额
- **5项量化准入条件**：最低GPA、挂科上限、志愿时长、综合得分、体测要求
- **互斥规则引擎**：任意两个奖项之间可配置互斥关系（如国家奖学金与校一等奖学金不可兼得），评选时自动检测并标记冲突

### 3. 自动评选引擎
- 一键运行，基于学生数据 + 奖项条件 + 互斥规则进行全量匹配
- 按照综合得分降序排列，取前 N 名符合条件者
- 输出每位候选人详细的**得分明细**（学业/德育/实践/体测/加分五维拆解）
- 对不合格候选人精确标注**拒绝原因**（如"GPA 3.45 < 3.50"、"挂科1门"、"与校一等奖学金互斥"）

### 4. 可视化工作台
- Dashboard 概览页展示学生总数、活跃奖项、评选名额、平均绩点等关键指标
- 评选结果页按奖项/学院/状态三维筛选，卡片展开查看详细得分条

## 技术亮点

- **纯前端状态驱动的评选引擎**：将复杂的多条件筛选 + 互斥冲突检测逻辑在浏览器端完成，无需后端即可运行完整评选流程。评选引擎核心约 70 行，覆盖 6 种拒绝场景和动态互斥检测。
- **组件化架构**：Layout（侧边栏导航）+ 4 个 Page 组件 + Context 全局状态管理，结构清晰，易于扩展为实际多用户系统。
- **类型安全**：全链路 TypeScript 严格类型定义，从 Student/Scholarship/EvaluationResult 数据模型到 Context API 均有类型约束。
- **快速部署**：集成 Vercel 配置，SPA 单页应用一键部署。

## 难点与解决方案

**难点 1：互斥规则如何在评选中生效？**

奖学金互斥不是简单的"A与B不能同时获得"，而是"A的获得者不能再获得B"。这意味着评选有**顺序依赖**——先评高优先级奖项，后评低优先级时需检查候选人是否已在此前轮次中获得了互斥奖项。

**解决**：评选引擎按奖项顺序迭代，维护一个全局结果集。每轮评选时，先通过基础条件过滤，再遍历互斥规则表，检查当前候选人是否已在结果集中出现且与当前奖项存在互斥关系。若存在，标注冲突原因并标记为不合格。

**难点 2：如何让评选结果可追溯？**

如果系统直接输出"张三获得国家奖学金"，教务老师无法验证公正性。

**解决**：每位候选人的评估结果携带完整元数据——五维得分明细、各项条件的判定结果、互斥冲突的具体奖项名称。所有判定逻辑透明化，评审可以展开任意一条结果追溯评分依据。

---

## 项目价值

- 将奖学金评选流程从**纯人工 → 规则化自动化**，减少人为误差
- 评选规则可配置、可复用，适配不同学年的政策变化
- 评选结果**可追溯、可解释**，满足教务审计要求
- 为后续接入学校教务系统数据库（学生成绩、志愿时长等）预留了清晰的接口边界
